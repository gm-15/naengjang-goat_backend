## 확정 ERD 요약

### 엔티티 목록

| 엔티티 | 테이블명 | 역할 |
|--------|---------|------|
| `User` | `users` | 점주 계정 |
| `Ingredient` | `ingredient` | 원재료 마스터 (구 RawMaterial) |
| `InventoryBatch` | `inventory_batch` | 입고 배치 — FIFO 관리 (구 Inventory 대체) |
| `Menu` | `menu` | 판매 상품 (구 Recipe) |
| `Recipe` | `recipe` | BOM — 메뉴-재료 연결 (구 RecipeItem) |
| `Order` | `orders` | 주문 이력 (구 SaleHistory) |
| `OrderItem` | `order_item` | 주문 상세 — N:M 해소 (구 SaleItem 재설계) |
| `MarketPrice` | `market_price` | KAMIS 시세 데이터 (구 PriceHistory) |
| `WasteLog` | `waste_log` | 폐기 오차 기록 — Phase 3 이후 선택 구현 |

### 엔티티 필드 정의

**User**
```
id           BIGINT PK AUTO_INCREMENT
username     VARCHAR(50) UNIQUE NOT NULL
created_at   DATETIME NOT NULL
```

**Ingredient** (구 RawMaterial)
```
id                 BIGINT PK AUTO_INCREMENT
user_id            BIGINT FK → users.id
name               VARCHAR(100) NOT NULL
base_unit          VARCHAR(10) NOT NULL        ← g / ml / 개
warning_threshold  DECIMAL(10,3)              ← 발주 알림 임계값
```

**InventoryBatch** (구 Inventory 대체)
```
id               BIGINT PK AUTO_INCREMENT
ingredient_id    BIGINT FK → ingredient.id
quantity         DECIMAL(10,3) NOT NULL       ← BigDecimal, 동시성 제어 핵심 대상
cost_per_unit    DECIMAL(10,2)               ← 입고 단가, 원가 분석용
inbound_date     DATE NOT NULL
expiration_date  DATE NOT NULL               ← FIFO 정렬 기준, D-3 알림 기준
created_at       DATETIME NOT NULL
```

**Menu** (구 Recipe)
```
id       BIGINT PK AUTO_INCREMENT
user_id  BIGINT FK → users.id
name     VARCHAR(100) NOT NULL
price    INT NOT NULL
```

**Recipe** (구 RecipeItem — BOM)
```
id                 BIGINT PK AUTO_INCREMENT
menu_id            BIGINT FK → menu.id
ingredient_id      BIGINT FK → ingredient.id
required_quantity  DECIMAL(10,3) NOT NULL    ← BigDecimal (구 Double)
unit               VARCHAR(10) NOT NULL      ← UnitConverter 동작 필수 필드
```

**Order** (구 SaleHistory)
```
id            BIGINT PK AUTO_INCREMENT
user_id       BIGINT FK → users.id
channel_type  ENUM('POS','DELIVERY','KIOSK') NOT NULL   ← 다채널 동시성 핵심
order_status  ENUM('COMPLETED','CANCELED') NOT NULL
total_amount  INT NOT NULL
created_at    DATETIME NOT NULL
```

**OrderItem** (구 SaleItem 재설계)
```
id          BIGINT PK AUTO_INCREMENT
order_id    BIGINT FK → orders.id
menu_id     BIGINT FK → menu.id
quantity    INT NOT NULL
unit_price  INT NOT NULL    ← 판매 시점 가격 스냅샷
```

**MarketPrice** (구 PriceHistory)
```
id               BIGINT PK AUTO_INCREMENT
ingredient_id    BIGINT FK → ingredient.id
retail_price     VARCHAR(20)      ← KAMIS 소매가
wholesale_price  VARCHAR(20)      ← KAMIS 도매가
unit             VARCHAR(20)
reported_date    DATE NOT NULL
```

**WasteLog** (Phase 3 이후 선택 구현)
```
id                 BIGINT PK AUTO_INCREMENT
ingredient_id      BIGINT FK → ingredient.id
log_date           DATE NOT NULL
theoretical_usage  DECIMAL(10,3) NOT NULL
actual_usage       DECIMAL(10,3) NOT NULL
deviation_reason   VARCHAR(255)
```

### 1-4. UnitConverter

```
global/util/UnitConverter.java

지원 변환
  WEIGHT: g ↔ kg  (× 1000)
  VOLUME: ml ↔ L  (× 1000)
  COUNT:  개 (변환 없음)

규칙
  - 모든 연산은 BigDecimal
  - 반올림 정책: HALF_UP
  - 단위 불일치 시 IllegalArgumentException("단위 변환 불가: g → ml")
  - 기준 단위(g, ml, 개)로 정규화 후 차감

사용 예시
  레시피: 마늘 50g 필요
  재고:   마늘 0.5kg 보유
  → UnitConverter.toBase("kg", 0.5) = 500g
  → 500g - 50g = 450g 잔량
```

## Phase 2 — LockStrategy 아키텍처 + 테스트

### 2-1. 전략 목적 정의

| 전략 | 목적 | 운영 여부 |
|------|------|---------|
| `NONE` | 락 없음 — 재고 붕괴 재현, 기준점 확보 | 테스트 전용 |
| `SPIN` | Redis SETNX 반복 시도 — 단순 구현, Redis 장애 시 무한루프 위험 | 테스트 전용 |
| `REDISSON` | Pub/Sub 기반 — CPU 오버헤드 낮음, 서킷 브레이커 탑재 | **최종 운영** |
| `PESSIMISTIC` | DB SELECT FOR UPDATE — Redis 없어도 동작 | Fallback 전용 |

**최종 운영 구조**: `REDISSON` + 서킷 브레이커 → Redis 장애 시 `PESSIMISTIC` 자동 Fallback

---

### 2-2. LockStrategy 패키지 구조

```
global/lock/
├── LockType.java
│     enum: NONE, SPIN, REDISSON, PESSIMISTIC
│
├── LockStrategy.java
│     interface: void executeWithLock(String key, Runnable task)
│
├── LockStrategyHolder.java
│     AtomicReference<LockStrategy> — 런타임 전략 교체
│     기본값: REDISSON (테스트 완료 후 고정)
│
├── LockStrategyFactory.java
│     LockType → 구현체 생성 및 반환
│
└── impl/
    ├── NoLockStrategy.java          락 없이 즉시 실행
    ├── SpinLockStrategy.java        Redis SETNX 5초 대기 루프
    ├── RedissonLockStrategy.java    @CircuitBreaker + fallbackToPessimistic()
    └── PessimisticLockStrategy.java SELECT FOR UPDATE + @Transactional
```

---

### 2-3. Resilience4j 서킷 브레이커

```yaml
# application.properties 추가
resilience4j:
  circuitbreaker:
    instances:
      redisLock:
        slidingWindowSize: 10
        failureRateThreshold: 50
        waitDurationInOpenState: 10s
        permittedNumberOfCallsInHalfOpenState: 3
```

```java
// RedissonLockStrategy.java
@CircuitBreaker(name = "redisLock", fallbackMethod = "fallbackToPessimistic")
public void executeWithLock(String key, Runnable task) {
    // Redisson 락 로직
}

private void fallbackToPessimistic(String key, Runnable task, Exception e) {
    log.warn("[Circuit Open] Redis 장애 → DB Lock 전환");
    pessimisticLockStrategy.executeWithLock(key, task);
}
```

**build.gradle 추가 의존성**
```groovy
implementation 'io.github.resilience4j:resilience4j-spring-boot3:2.1.0'
implementation 'org.springframework.boot:spring-boot-starter-aop'
```

### 2-5. 테스트 계획 (3단계 — 로컬 실행)

> 발표 시에는 결과 수치표 + 시연 영상으로 대체. 별도 프론트 화면 불필요.

**단계 1 — 정합성 붕괴 재현 (NONE 전략)**

```
목적:  락 없을 때 데이터가 실제로 터지는 것을 수치로 확인
조건:  재고 500 세팅, NONE 전략 고정
방법:  CountDownLatch 500 스레드 동시 차감 요청 (각 1씩)
확인:  최종 재고 수치 출력 (예상: 음수)
기록:  최종 재고, 오차량 → 수치표에 기입
```

**단계 2 — 전략별 정합성 + 성능 비교**

```java
// ConcurrencyTest.java
@ParameterizedTest
@EnumSource(LockType.class)
void 전략별_동시성_테스트(LockType lockType) {
    lockStrategyHolder.set(lockStrategyFactory.create(lockType));
    // 재고 500 초기화
    // CountDownLatch(500) 동시 실행
    // 최종 재고 출력
}
```

```
목표 수치표 (실측 후 기입)
┌────────────┬──────────┬──────────┬──────────┬────────────────┐
│ 전략       │ 동시요청  │ 정합성오차 │ 평균지연  │ Redis 장애대응  │
├────────────┼──────────┼──────────┼──────────┼────────────────┤
│ NONE       │ 500      │ 측정값   │    -     │ 서비스 중단     │
│ SPIN       │ 500      │ 0        │ 측정값   │ 서비스 중단     │
│ REDISSON   │ 500      │ 0        │ 측정값   │ 자동전환 ✅    │
│ PESSIMISTIC│ 500      │ 0        │ 측정값   │ 자동전환 ✅    │
└────────────┴──────────┴──────────┴──────────┴────────────────┘
```

**단계 3 — 카오스 엔지니어링 (Redis 강제 종료)**

```
목적:  서킷 브레이커 → PESSIMISTIC Fallback 실제 동작 확인
방법:
  1. REDISSON 전략 활성 상태에서 요청 처리 중
  2. Redis 서버 강제 종료 (docker stop redis)
  3. 서킷 브레이커 OPEN 여부 확인
  4. PESSIMISTIC으로 자동 전환 후 서비스 무중단 확인
  5. 로그 "[Circuit Open] Redis 장애 → DB Lock 전환" 출력 확인
기록:  영상 촬영 (발표 시연 영상용)
```
