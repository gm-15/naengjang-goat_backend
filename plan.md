# 냉장GOAT — 사용자 중심 재편 계획

> 작성일: 2026-04-05
> 근거: 현장 인터뷰 2건 + Gemini UX 리뷰 + 글로벌 외식업 통계
> 상태: **검토 중 — 구현 금지**

---

## 핵심 전환 선언

> "엔진은 람보르기니인데 운전석이 없다." — Gemini UX 리뷰

현재 시스템은 기술적 정합성(동시성, 분산락)은 증명됐으나
사장님이 쓸 이유가 없다. 재편 방향:

**"사장님의 30분을 아껴주는 앱"**

사장님은 데이터 정합성이 아니라 **걱정과 수고를 덜어주는 대가**로 월 3만원을 낸다.
(근거: 용인 막창점 인터뷰)

---

## 현장 근거 요약

### 인터뷰 Pain Point
| 문제 | 출처 | 빈도 |
|------|------|------|
| 재료 부족으로 판매 불가 | 송탄점 인터뷰 | 월 3~6건 |
| 발주 미스로 다른 매장에서 재료 빌림 | 용인 막창점 인터뷰 | 실제 발생 |
| 거래처 5곳에 개별 문자/전화 발주 | 송탄점 인터뷰 | 매주 |
| 원가율 수동 계산 (40~50% 직접 파악) | 용인 막창점 인터뷰 | 상시 |
| 재고 조사 눈대중 | 양쪽 공통 | 격일 |
| 앱 3개 분산 사용 | 송탄점 인터뷰 | 상시 |

### 통계 근거
- 식재료 **4~10% 폐기** 손실 (글로벌 외식업 통계)
- 수기 재고 확인 **주당 3~5시간** 소요
- 소상공인 음식점 **85.7%가 근로자 평균임금 미만** 수익 → 원가 1%도 민감

## 릴리즈별 범위

---

### 🔴 Demo (현재 진행 중 — 프론트 대기 중)

목표: 가게에서 실제로 주문을 받을 수 있는 최소 상태

**구현할 API**
| API | 설명 | 상태 |
|-----|------|------|
| `POST /orders` | POS 주문 접수 + 재고 자동 차감 | ✅ 존재 |
| `GET /menus` | 메뉴 목록 조회 | ✅ 존재 |
| `GET /ingredients/{id}/batches` | 재고 현황 조회 | ✅ 존재 |
| `POST /ingredients` | 재료 등록 | ❌ 신규 |
| `POST /menus` | 메뉴 등록 | ❌ 신규 |
| `POST /ingredients/{id}/batches` | 입고 등록 | ❌ 신규 |
| `POST /menus/{id}/bom` | BOM 연결 | ❌ 신규 |

**인증**: MockAuthFilter 유지 (X-User-Id 헤더)
**범위 외**: 알림, 발주, 원가 계산 — V1으로

---

### 🟡 V1 (데모 이후 — 실제 가게 적용)

목표: 사장님이 "이거 덕분에 퇴근 30분 빨라졌다"고 느끼는 상태

#### V1-1. 재고 경고 알림
근거: 용인 막창점 "카카오톡 알림 선호", 송탄점 월 3~6건 판매 불가
- `warningThreshold` 필드 이미 존재 (Ingredient 엔티티)
- 재고 < warningThreshold 시 카카오 알림톡 발송
- 유통기한 D-3 임박 배치 알림
- 구현: 카카오 알림톡 API 연동 or 테스트용 Mock 서버

#### V1-2. 원가율 자동 계산
근거: 용인 막창점 "원가율 40~50% 직접 계산, 계란값 올라 가격 인상 고민"
- BOM × 입고단가 = 이론 원가 자동 계산
- KAMIS 시세 연동 → "이 메뉴 팔 때마다 500원 손해" 형태로 표현
- 원가율 경고 임계값 초과 시 알림
- 기술 과제: KAMIS 가격 변동 이벤트 시에만 연관 메뉴 재계산 (성능)

#### V1-3. 발주 관리
근거: 용인 막창점 "다른 매장에서 빌린 경험", 송탄점 "5곳 개별 문자"
- 거래처 등록 (이름, 연락처, 담당 품목)
- 재고 임계값 도달 시 발주 문자 초안 자동 생성
- 발주 이력 관리
- 구현: SMS 또는 카카오 메시지 API

#### V1-4. 입고 등록 간소화
근거: Gemini 리뷰 "입력하는 게 제일 귀찮아, 3일 만에 앱 지울 거야"
- 이전 발주 이력 기반 '퀵 입고' — 수량만 확인하면 완료
- 거래처별 품목 자동 매칭
- 입고 시 배치 자동 생성 + FIFO 적용

#### V1-5. 판매 현황 대시보드
근거: 용인 막창점 "손해 메뉴(추억의 도시락) 알고 있으나 수치 없음"
- 일/주간 메뉴별 판매량
- 메뉴별 원가율 + 예상 수익
- 채널별(POS/배달/키오스크) 매출 비교

### 🟢 V2 (검증 후)

목표: 사장님이 "이게 없으면 장사 못 하겠다"고 느끼는 상태

- 영수증 OCR 자동 입고 (입력 자동화)
- AI 발주량 추천 (소진 속도 기반)
- ROI 시뮬레이션 ("이 앱으로 한 달 X만원 절약")
- JWT 인증 복구 (운영 환경 보안)
- nGrinder HTTP 부하 테스트

## 기술 아키텍처 유지 사항

사장님은 모르지만 면접관이 보는 것들 — 건드리지 않음

| 기술 | 역할 | 상태 |
|------|------|------|
| Redisson + Circuit Breaker | 다채널 동시 주문 정합성 | ✅ 완료 |
| FIFO (InventoryBatch) | 유통기한 순 소진 | ✅ 완료 |
| BigDecimal | 원가 소수점 정밀도 | ✅ 완료 |
| KAMIS Batch | 시세 수집 | ✅ 완료 (스케줄러 복구 필요) |
